# 開発ルール（CONTRIBUTING）

このリポジトリでの作業は、以下のルールに従って行う。コラボレーターが本人のみの場合でも例外なく適用する。

## 1. Issue駆動開発

- 機能追加・修正・改善を行う前に、必ずGitHub Issueを起票する。
- Issueがない変更（誤字修正など極小なものを除く）は原則行わない。
- Issueには背景（なぜ必要か）と完了条件（何ができればOKか）を書く。

## 2. ブランチ運用（GitHub Flow）

- `master` は常にリリース可能な状態を保つ。`master` への直接pushは行わない（GitHub側でも禁止設定済み）。
- 作業は必ず `master` からブランチを切って行う。

### ブランチ命名規則

```
<種別>/#<issue番号>-<概要（英語・ハイフン区切り）>
```

| 種別 | 用途 |
|---|---|
| `feature/` | 新機能の追加 |
| `fix/` | 不具合修正 |
| `docs/` | ドキュメントのみの変更 |
| `chore/` | ビルド・設定・依存関係などの雑務 |
| `refactor/` | 挙動を変えないコード整理 |

例: `feature/#12-add-login`, `fix/#15-card-delete-bug`

## 3. コミットメッセージ規約（Conventional Commits）

```
<type>: <変更内容（短く）>
```

| type | 用途 |
|---|---|
| `feat` | 新機能 |
| `fix` | 不具合修正 |
| `docs` | ドキュメントのみの変更 |
| `refactor` | 挙動を変えないコード整理 |
| `test` | テストの追加・修正 |
| `chore` | ビルド・設定・依存関係などの雑務 |

例: `feat: カードのドラッグ&ドロップ並び替えを追加`

## 4. プルリクエスト（PR）フロー

1. Issueを起票する。
2. Issueに対応するブランチを作成する。
3. 作業し、Conventional Commitsでコミットする。
4. `master` へのPRを作成し、本文に `Closes #<issue番号>` を記載してIssueと紐付ける。
5. PRテンプレートのチェックリストに従ってセルフレビューする。
6. **Squash and merge** でマージする（マージ方法はSquashのみ許可）。
7. マージ後、ブランチは自動削除される。

## 5. ブランチ保護

`master` ブランチには以下を設定済み（管理者にも適用される）。

- PRを経由しない直接pushを禁止
- force pushを禁止
- ブランチの削除を禁止

CI（テスト・lintの自動実行）を導入した場合は、必須ステータスチェックとして本ルールに追記する。
